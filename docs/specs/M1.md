# M1 — Mobile foundation: shared-backend auth, both surfaces, diagnostics

> Status: **frozen** · Written: 2026-06-26
> Frozen once coding begins. Deviations go in the milestone plan or an ADR.

## Goal

Stand up the native mobile companion to the Clearview Savings web app: an Expo /
React Native app that authenticates against the **same Supabase backend**, ships
both the caregiver and patient surfaces, and includes an in-app backend test
harness so connectivity can be verified on the device without external tooling.
This milestone is the foundation every later feature builds on.

## Scope

- Expo + expo-router + TypeScript project scaffold.
- Supabase client wired to the shared project via `EXPO_PUBLIC_*` keys, with
  persistent sessions and token auto-refresh (no idle timeout).
- Caregiver sign-in using existing web credentials.
- Caregiver mode: patient list, patient detail (accounts + balances),
  persistent "Caregiver mode" banner, sign-out.
- Patient mode (entered from caregiver mode): "Your Accounts" home with
  Available Balance + Direct Deposit Pending, and account detail with Recent
  Transactions. Large-type, calm, real-bank vocabulary.
- Always-present regulatory disclosure footer on every screen.
- In-app **Backend diagnostics** screen: env, auth session, RLS read
  round-trip, token freshness.
- The multi-session handoff/re-anchor system (this docs tree + scripts +
  memory).

## Non-goals

- No mutations that affect balances (deposit-code redemption, transfers,
  scheduled-deposit materialization). Those stay in the web app for M1.
- No separate patient login. Patient view runs inside the caregiver's session.
- No push notifications, no offline cache, no biometric unlock (later
  milestones may revisit).
- No app-store build/submission pipeline (EAS) in M1.

## Acceptance criteria

1. A caregiver can sign in with their web credentials and see only their own
   patients (RLS-scoped).
2. Tapping a patient shows that patient's accounts and balances.
3. "Switch to patient view" (with inline confirmation) opens the patient bank
   view for that patient.
4. Patient screens follow the patient UX rules (≥18px body, 28px+ headings, no
   modals, calm fallback on error, real-bank vocabulary and titles).
5. The disclosure footer appears on every screen.
6. The Diagnostics screen runs all checks and reports pass/fail with latency.
7. `npm run typecheck` passes.
8. The handoff/re-anchor system exists and `scripts/re-anchor.sh` runs.

## Open questions

- Should later milestones add a mobile-safe deposit-code redemption flow, or
  keep all writes in the web app? (Deferred — see Non-goals; revisit in a spec.)
- Biometric (Face ID / fingerprint) gate before entering caregiver mode?

## Risks

- **Schema drift** between this app's `lib/queries.ts` types and the web Drizzle
  schema. Mitigation: CLAUDE.md mandates keeping them in sync; types are narrow
  and centralized.
- **Expo SDK / RN version skew** on `npm install`. Mitigation: pinned ranges in
  `package.json`; run `npx expo install --fix` if the resolver complains.
