# M3 — Implementation plan

Plan for delivering the M3 spec (`docs/specs/M3.md`): full caregiver parity
(minus MFA management + check/workbook generation) + a denser patient view.
Spans both repos. Built in 3 phases with a rebuild/test between each.

## Approach

Reuse the M2 pattern for every write: a thin `app/api/m/*` route in
`clearview-savings` that authenticates the caregiver bearer token
(`requireApiPatient`), validates with a shared Zod schema, calls a helper
extracted from the existing Server Action, and the action becomes a thin caller
(web behavior unchanged; web tests stay the regression net). Mobile calls them
through `lib/api.ts`. Reads stay PostgREST-under-RLS (one new `audit_log` grant).

## Phase 1 — Caregiver writes (web endpoints + mobile screens)

### Web (`clearview-savings`) — extract helper + add endpoint per action
1. **Add patient** — extract `addPatientAction` core → `lib/patients/add-patient.ts`
   (create patient + auto "Checking" account + audit, in a txn). `POST /api/m/patients`.
2. **Add account** — extract `addAccountAction` → `lib/accounts/add-account.ts`.
   `POST /api/m/accounts`.
3. **Rename account** — extract `renameAccountAction` →
   `lib/accounts/rename-account.ts`. `POST /api/m/accounts/rename`.
4. **Scheduled deposits** — extract add/toggle/delete →
   `lib/scheduled-deposits/manage.ts`. `POST /api/m/scheduled-deposits`,
   `…/toggle`, `…/delete`.
5. **Patient settings** — extract `updatePatientSettingsAction` →
   `lib/patients/update-settings.ts`. `POST /api/m/patient-settings`.
   - Each: auth → shared Zod → `requireApiPatient` ownership → helper → JSON.
   - Co-located helper tests (pg-mem) mirroring `manual-adjustment.test.ts`.
   - Verify: web `pnpm typecheck && lint && test` green (no behavior change).

### Mobile (`clearview-savings-mobile`)
6. `lib/api.ts` — typed methods for the new endpoints.
7. `lib/queries.ts` — `listScheduledDeposits(patientId)` (all, with status),
   inline `listTransactions` already exists.
8. Patients screen: **Add a patient** form.
9. Patient detail: **Add savings account**, **rename account**, **scheduled
   deposits** (list + add + pause/resume + delete), **patient settings**,
   **inline recent transactions** per account.
10. Demo mode: sample data + "changes aren't saved" guards for the new writes.
   - Verify: mobile `npm run typecheck && lint`; web export bundles.

## Phase 2 — Caregiver shell + audit log

11. **Top bar** on every caregiver screen: Settings + Sign out (keep the
    caregiver-mode banner). Likely a shared header in `(caregiver)/_layout.tsx`.
12. **Settings screen** (`app/(caregiver)/settings.tsx`): signed-in email, Sign
    out, and a link to manage 2FA on the web app. (No MFA management on mobile.)
13. **Audit log** (`app/(caregiver)/patient/[id]/audit.tsx` or a tab): read
    `audit_log` via PostgREST. Add `grant select on audit_log to authenticated`
    in `supabase/policies.sql` (re-verify RLS first) + apply to the DB.
14. **Footer** links (About/Privacy/Terms/Security) → open the web pages.

## Phase 3 — Denser patient view + polish

15. Redesign the patient accounts home: greeting, richer accounts, pending
    notice, Deposit-a-Check, a calm Security Reminder panel — within patient UX
    rules. No clinical/"demo" wording; no decorative nav implying missing
    capabilities.
16. A clearer loading/transition indicator between pages.

## Cross-repo contract

Same as M2: mobile → `EXPO_PUBLIC_API_BASE_URL` with `Authorization: Bearer
<token>`; server verifies identity + ownership; client never sends trusted ids.
New routes live under `app/api/m/*`.
