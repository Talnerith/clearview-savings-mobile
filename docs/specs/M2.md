# M2 — Live caregiver auth + the shared-backend write pattern

> Status: **frozen** · Written: 2026-06-27 · Frozen: 2026-06-27
> Frozen — coding has begun. Deviations go in the milestone plan or an ADR.

## Goal

Make the mobile app usable against the **real** shared Supabase backend (not
just demo mode) for a caregiver, and establish — end to end — the architecture
that lets mobile perform balance-affecting actions **without duplicating
business logic or shipping any secret key**. Two things block this today: (1)
the project enforces Cloudflare Turnstile on Auth, and the mobile sign-in sends
no captcha token, so every real login is rejected; (2) every web mutation is a
Next.js Server Action that React Native cannot call. M2 fixes auth and proves
the write pattern on a first, representative slice of features. The remaining
caregiver/patient write features ride the same rails in M3+.

This milestone spans **two repos**: `clearview-savings-mobile` (this app) and
`clearview-savings` (the web app, where the shared endpoints live). The web
app's balance logic is already extracted into reusable helpers
(`lib/deposit-codes.ts` → `redeemCode`, `lib/transfers/transfer.ts` →
`performTransfer`), which the shared endpoints reuse rather than reimplement.

## Scope

### A. Real caregiver authentication (mobile)
- Render a **Turnstile widget** on the mobile sign-in and pass its token as
  `captchaToken` to `supabase.auth.signInWithPassword`. **Native is the M2
  target** (the Android APK is the primary distribution): use a webview-backed
  Turnstile on native; the same `captchaToken` seam serves Expo-web with the JS
  widget. (Decision 2026-06-27.)
- Handle the **MFA challenge** (AAL1 → AAL2): if the caregiver has TOTP enabled,
  prompt for the 6-digit code and call `supabase.auth.mfa.challenge` + `verify`.
- Graceful, calm error states reusing the existing `<Notice>` pattern
  (captcha-rejection, bad-credentials, MFA-required).

### B. The shared-backend write pattern (web app + mobile)
- Add **JWT-authenticated API route handlers** in `clearview-savings`
  (e.g. under `app/api/m/…`) that: verify the caller's Supabase access token,
  resolve the caregiver, enforce the same ownership checks the Server Actions
  use, call the existing shared helper, and write to the audit log.
- Extract any inline Server Action logic that isn't already a helper into a
  shared module both the web Server Action and the new route handler call —
  **single source of truth, no balance logic in two places**.
- A small typed **mobile API client** (`lib/api.ts`) that attaches the current
  session's access token and calls these endpoints.

### C. First write slice (proves the pattern end to end)
- **Patient: deposit-code redemption** — the patient "Deposit a Check" flow
  (`redeemCode`), wired through a shared endpoint.
- **Caregiver: manual transaction + transfer** — the two core balance-affecting
  caregiver actions (`manualAdjustmentAction` logic + `performTransfer`).
- Each writes an audit-log entry identical to the web path.

## Non-goals

- **No secret keys in the mobile bundle, ever** — no `SUPABASE_SECRET_KEY`,
  `RESEND_API_KEY`, `ADMIN_EMAIL`, `TURNSTILE_SECRET_KEY`, cron/webhook secrets.
  Auth emails and admin/ops notifications continue to fire server-side from the
  same Supabase project + web deployment; mobile triggers them implicitly.
- **Web-only / ops surfaces are out of parity entirely**: PDF check/workbook
  printing, the `/admin` dashboard, the cron daily-digest, and the
  email-hook/Sentry webhooks. (Decision: "pragmatic parity".)
- **Remaining caregiver/patient writes deferred to M3+**: add patient / add
  account, scheduled deposits (create/pause/delete), check & workbook code
  generation, patient settings/rename, audit-log viewing. They reuse the M2
  rails; they are not built in M2.
- **MFA management deferred**: M2 handles the TOTP *challenge* at sign-in;
  enrollment, disable, and recovery-code management stay on the web app for now.
- No push notifications, offline cache, or biometric unlock.

## Acceptance criteria

1. A real caregiver (Turnstile-enforced project, no MFA) signs in on mobile and
   reaches "Your patients" with live data — the smoke test that M1 could not
   complete.
2. A caregiver **with** TOTP enabled is prompted for their code and completes
   sign-in (AAL2) on mobile.
3. The Diagnostics screen, signed in for real, shows the live Supabase URL and a
   passing RLS read round-trip with latency (not "(not set)" / demo).
4. From mobile, redeeming a valid deposit code posts the transaction and updates
   the balance, going through the shared endpoint — verified against the same
   row the web app would have written, with a matching audit-log entry.
5. From mobile, a manual transaction and an account-to-account transfer apply
   correctly and write audit entries identical to the web path.
6. A crafted `patientId`/`accountId` the caregiver doesn't own is rejected by the
   endpoint (ownership enforced server-side, not trusted from the client).
7. No secret-key env var is referenced anywhere in the mobile bundle; only
   `EXPO_PUBLIC_*` values.
8. Web app: `npm run typecheck` + lint clean and its existing tests still pass
   (the helper extraction did not change web behavior). Mobile: `npm run
   typecheck` + lint clean.

## Resolved decisions (2026-06-27)

- **Native real-auth in M2.** Build Turnstile auth for the Android APK via a
  webview-backed widget; the `captchaToken` seam also serves Expo-web. Adds a
  dependency + on-device testing — accepted because the APK is the point.
- **First write slice = deposit redemption + manual transaction + transfer.**
  Both personas, across the shared-endpoint pattern. Remaining caregiver/patient
  writes deferred to M3+.

## Open questions

- **Webview Turnstile lib choice.** Pick the specific native webview-Turnstile
  approach during the plan (a `react-native-turnstile`-style lib vs a hand-rolled
  `react-native-webview` host). Verify it resolves a token Supabase accepts.
- **Endpoint host + CORS.** The shared endpoints live on the web app's
  deployment (Vercel). Confirm the mobile app should call the production web
  origin, and add the mobile origin(s) to CORS for web-export. A new
  `EXPO_PUBLIC_API_BASE_URL` is likely needed.
- **Auth-token verification in the route handlers.** Verify the Supabase JWT via
  `supabase.auth.getUser(token)` on each request (simple, one network hop) vs
  local JWKS verification (faster, more setup). Default to `getUser` for M2.
- **Is the test caregiver MFA-enabled?** Determines whether AC-2 can be tested
  this milestone or needs a TOTP enrollment on web first.

## Risks

- **Web-app refactor regressions.** Extracting Server Action logic into shared
  helpers could change web behavior. Mitigation: keep the Server Action as a
  thin caller of the helper; rely on the web app's existing tests + typecheck;
  do the extraction as small, reviewed steps.
- **Two-repo coordination.** M2 touches `clearview-savings` and
  `clearview-savings-mobile`; both are public. Mitigation: land the web
  endpoints first (additive, behind no flag — they're new routes), then the
  mobile client; document the cross-repo contract in the M2 plan.
- **Duplicated validation drift.** The endpoint must enforce the same Zod
  validation + ownership as the Server Action. Mitigation: share the Zod schema
  and the ownership helper (`getPatientForCaregiver`) between both callers.
- **Native Turnstile friction.** Webview captcha can be finicky on RN.
  Mitigation: if native proves slow, fall back to web-first per the Open
  question and keep native behind the same `captchaToken` seam.
- **Auth-state surface area.** Adding the MFA challenge changes the auth state
  machine (`lib/auth.tsx`). Mitigation: model AAL explicitly; demo mode and the
  no-idle-timeout rules must stay intact.
