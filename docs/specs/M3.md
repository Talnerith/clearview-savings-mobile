# M3 — Full caregiver parity + denser patient view

> Status: **frozen** · Written: 2026-06-27 · Frozen: 2026-06-27
> Frozen — coding has begun. Deviations go in the milestone plan or an ADR.

## Goal

Bring the mobile app to feature parity with the web **caregiver** experience and
give the **patient** view a denser, more bank-like layout. M2 proved the
shared-endpoint write pattern on three actions; M3 extends it across the
remaining caregiver writes and adds the caregiver shell (top-bar Settings +
Sign out) and richer reads. On-device smoke testing showed the M2 mobile app,
while functional, is missing most of what the web caregiver can do — M3 closes
that gap. **MFA management stays on the web app** (decision 2026-06-27); the
mobile Settings screen points there for 2FA.

Built in reviewable phases with a rebuild/test between each (decision: spec
first, batches).

## Scope

### A. Caregiver writes — new `app/api/m/*` endpoints (reuse web helpers) + mobile UI
- **Add a patient** (creates the patient + its auto "Checking" account).
- **Add a savings account**; **rename an account**.
- **Scheduled deposits**: add, pause/resume, delete.
- **Patient settings**: display name, font size, locale (currency follows
  locale).

(Check & workbook code **generation** is intentionally NOT here — see Non-goals.)

### B. Caregiver reads (PostgREST under RLS; add SELECT grants where needed)
- **Inline recent transactions** per account on the patient detail.
- **Audit log** screen (needs `grant select on audit_log`).

### C. Caregiver shell
- **Top bar** on every caregiver screen: **Settings** + **Sign out** (keep the
  "caregiver mode" banner).
- **Settings screen**: signed-in email, Sign out, and a clear link to manage
  two-factor authentication on the web app. (No MFA management on mobile.)
- **Footer** links (About / Privacy / Terms / Security) that open the web pages.

### D. Patient view density
- Greeting, richer accounts layout, pending-deposit notice, Deposit-a-Check
  entry, and optional calm bank-like panels (e.g. a Security Reminder) — all
  within the patient UX rules (large type, no clinical/"demo" wording, calm
  fallbacks, one primary action emphasis).

### E. UI polish
- A clearer loading/transition indicator between pages (per-screen spinners
  already exist). Button press states already work on native; hover is web-only.

## Non-goals

- **MFA management on mobile** (enroll / disable / regenerate recovery codes) —
  stays on the web app. Mobile Settings links there.
- **Check & workbook codes — generation, printing, AND the codes list — stay on
  the web app.** Generating a code on mobile makes no sense when the check /
  workbook must be printed (web-only, `@react-pdf`). Patients still **redeem**
  any code through the existing mobile Deposit-a-Check flow (built in M2). A
  read-only mobile list of codes was dropped for the same reason; easy to add
  back later if wanted.
- Admin dashboard, cron digest, email/Sentry webhooks — never in scope.
- **No secret keys in the mobile bundle** — unchanged. New writes go through the
  server with the caregiver's JWT; new reads are RLS-scoped PostgREST.
- Decorative non-functional bank nav (Bill Payments / Statements / Tax Documents
  / Order Checks) — only if it adds to the patient illusion without implying real
  capability (see Open questions).

## Acceptance criteria

1. A caregiver can **add a patient** on mobile; it appears with a Checking
   account.
2. **Add a savings account**; **rename** either account; both reflect live.
3. **Add a scheduled deposit**, then **pause/resume** and **delete** it; the
   patient sees "Direct Deposit Pending" within the pending window.
4. **Update patient settings** (name / font / locale); the patient view reflects
   them (currency follows locale).
5. **Audit log** lists the caregiver's actions.
6. **Recent transactions** show inline per account on the patient detail.
7. Every caregiver screen has **Settings + Sign out**; Settings shows the email,
   signs out, and links to web 2FA.
8. The **patient view** is denser/bank-like and still passes the patient UX
   rules (no clinical/"demo" wording anywhere patient-visible, calm fallbacks,
   large type, real-bank vocabulary/titles).
9. No secret env referenced in the mobile bundle; **both repos typecheck + lint
   clean**; the web app's existing test suite stays green; new endpoint helpers
   get co-located tests.

## Open questions

- **Audit log read:** PostgREST + a new SELECT grant on `audit_log` (consistent
  with M2 reads) vs a GET endpoint. Default: PostgREST + grant (re-verify RLS
  scoping before granting, same caregivers-subquery dependency that bit M2).
- **Patient-view decorative panels:** include non-functional bank chrome
  (Messages / Security Reminder / Quick Links) to match the web feel, or keep
  only functional elements? Leaning: a calm Security Reminder + pending/accounts
  only; skip anything implying a capability the app doesn't have.
- **Demo mode:** extend demo data + "changes aren't saved" guards to the new
  caregiver writes, and the denser patient view, so the public web demo stays
  walkable.

## Risks

- **Endpoint surface area** (~8 new routes). Mitigation: each reuses an extracted
  web helper + the existing `requireApiPatient` auth/ownership; web tests guard
  behavior; extract helpers in small reviewed steps (as M2 did for manual-adj).
- **New PostgREST grant** (`audit_log`). Mitigation: confirm RLS enabled +
  policy present (read-only check) before granting; SELECT only.
- **Patient-view redesign** must not break the therapeutic illusion or the
  patient UX rules. Mitigation: keep CLAUDE.md rules front-and-center; no
  clinical/"demo" wording; calm fallbacks.
- **Scope size.** Mitigation: phased delivery — (1) caregiver writes + inline
  transactions, (2) shell + audit log, (3) denser patient view + polish — with a
  rebuild/test between phases.
