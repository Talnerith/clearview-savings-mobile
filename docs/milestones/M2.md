# M2 — Implementation plan

Plan for delivering the M2 spec (`docs/specs/M2.md`): live caregiver auth +
the shared-backend write pattern. Records *how* M2 is built and any deviations
from the frozen spec. **Two repos:** `clearview-savings` (web, endpoints) and
`clearview-savings-mobile` (this app, client).

## Approach

Build in dependency order so each step is independently verifiable and the
web-app changes are additive (new routes, no behavior change to existing flows):

1. Web app: extract logic into shared helpers + expose authenticated endpoints.
2. Mobile: real auth (Turnstile + MFA challenge).
3. Mobile: API client + the three write features.
4. Verify end-to-end on web-export and on a device (APK), update docs/memory.

## Steps

### Phase 1 — Web app: shared endpoints (`clearview-savings`)
1. **Cross-repo contract doc** — short note in the web repo describing the
   `app/api/m/*` surface, auth header, and shared origin, so the web repo's own
   handoff captures it.
2. **JWT auth helper** — `requireCaregiverFromToken(req)` that reads the
   `Authorization: Bearer <supabase access token>`, calls
   `supabase.auth.getUser(token)`, and returns the caregiver (401 otherwise).
3. **Extract remaining inline logic into helpers** — `manualAdjustment` logic
   into a shared module alongside the existing `redeemCode`
   (`lib/deposit-codes.ts`) and `performTransfer` (`lib/transfers/transfer.ts`).
   Keep each Server Action a thin caller of its helper (no web behavior change).
4. **Endpoints** (each: auth → shared Zod validate → ownership check
   (`getPatientForCaregiver`) → helper → audit-log write → JSON result):
   - `POST /api/m/deposit/redeem`
   - `POST /api/m/transactions/manual`
   - `POST /api/m/transfers`
5. **CORS** for the Expo-web origin(s); native has no CORS. Verify web tests +
   typecheck + lint still green.

### Phase 2 — Mobile: real auth
6. **Turnstile widget** (`components/Turnstile.tsx`) — webview-backed on native,
   JS widget on web; emits a token via callback. Reads
   `EXPO_PUBLIC_TURNSTILE_SITE_KEY`.
7. **Sign-in wiring** — `app/(auth)/sign-in.tsx` + `lib/auth.tsx`: pass
   `captchaToken` into `signInWithPassword`; surface captcha-rejection calmly.
8. **MFA challenge** — extend the auth state machine for AAL1→AAL2: detect
   MFA-required, add a TOTP code screen (`app/(auth)/challenge.tsx`) calling
   `supabase.auth.mfa.challenge` + `verify`. Demo mode + no-idle-timeout intact.

### Phase 3 — Mobile: API client + write features
9. **API client** — `lib/api.ts`: attaches the session access token, posts to
   `EXPO_PUBLIC_API_BASE_URL`, typed request/response, calm error mapping.
10. **Deposit redemption** — wire the patient "Deposit a Check" flow to
    `/api/m/deposit/redeem` (new patient deposit screen/flow).
11. **Manual transaction + transfer** — caregiver actions on the patient detail
    screen, calling `/api/m/transactions/manual` and `/api/m/transfers`.

### Phase 4 — Verify + document
12. Web-export walkthrough + on-device APK walkthrough (real sign-in, MFA,
    redeem, manual txn, transfer); confirm audit entries match the web path and
    ownership rejection works. Update `docs/milestones/M2-progress.md` and the
    relevant memory.

## Cross-repo contract (summary)

- **Base URL:** `EXPO_PUBLIC_API_BASE_URL` → the web app's deployment origin.
- **Auth:** `Authorization: Bearer <supabase session access token>`; server
  verifies via `supabase.auth.getUser(token)` and enforces ownership. The client
  never sends `caregiverId`/`patientId` as trusted identity — a crafted id
  returns 403/zero rows.
- **Endpoints:** `POST /api/m/deposit/redeem`, `/api/m/transactions/manual`,
  `/api/m/transfers`. JSON in/out; shared Zod schemas with the web actions.
